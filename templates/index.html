                        📦 GGUF Configuration Guide
                    

                    

                        

                            vLLM supports native GGUF model execution, which is highly compatible across Nvidia GPU generations.
                        

                        

                            🚀 GPU Compatibility:
                            
    
                                
  • Turing & Newer (RTX 20-series+): Full support for native kernels, PagedAttention, and optimal execution speeds.
    • 
                                
  • Pascal & Older (GTX 10-series): Works fine, but runs slower as modern optimized attention kernels require compute capability 7.0+.
    • 
                            

                        

                        

                            ⚙️ Setup Requirements:
                            
    
                                
  • Point to the local file path (e.g., /models/model.gguf) as the Model ID.
    • 
                                
  • Specify a standard HuggingFace repo ID under Tokenizer Repository (vLLM requires the repo config files to parse vocabulary).
    •

Speculative Decoding

                        

                            Accelerate inference speeds by predicting multiple tokens per iteration. vLLM supports both Assisted drafting models and built-in or custom Draft Heads/MTP models.
                        

                        
                        

                            
                            
                        


                        

                            
                            

                                
                                
                                
                                
                            

                        


                        

                            
                            
                        


                        
                        

                            

                                💡 Speculative Decoding Smart Advisor
                            

                            
                            

                                Select a speculative mode above to view advice.
                            


                            

                                

                                    1. Drafting Model (Assisted/Speculative Model)
                                    
A smaller, faster secondary model runs ahead of the main model, speculating tokens. The main model verifies them in parallel. Note: Not recommended for small quantized base models (like 4B-AWQ) because the overhead of running a separate model exceeds the speed benefit.

                                

                                

                                    2. MTP (Multi-Token Prediction)
                                    
Instead of generating one token at a time, the model backbone is trained to predict several future tokens simultaneously in a single forward pass, drastically increasing inference throughput.

                                

                                

                                    3. Header Model (or Draft Head)
                                    
A specialized, lightweight neural network layer (e.g. Medusa/EAGLE) attached directly to the main architecture. It acts exclusively as the drafter without launching a separate model. In vLLM, configure as [draft] or point to the Medusa repo.

Usable vs. Maximum Context

                
            

            

                

                    You may notice a discrepancy between the Verified Usable Context (e.g., 32.8k) and the logical KV Cache capacity (e.g., 84.6k).
                

                

                    
    
                        
  • 
                            Single-Agent Context Limit (max-model-len): 
                            A single inference request is strictly limited by the model's native context length (e.g. 32,768). A single agent conversation can never exceed this limit.
                        
    • 
                        
  • 
                            Logical KV Cache Pool Maximum: 
                            The larger number (e.g. 84.6k) represents the total concurrent capacity of physical KV cache pages pre-allocated on VRAM. This is shared across all parallel sequences (concurrency seats, e.g. 16).
                        
    • 
                        
  • 
                            Windows Host Paging & WDDM Overhead: 
                            On Windows systems, if vLLM's memory allocation (utilization) is configured too high, the OS's WDDM driver swaps active VRAM pages to Host RAM. This degrades performance below the stable benchmark threshold (15.0 tokens/s), causing the verification to report a lower stable limit.
                        
    • 
                    

                

                

                    💡 Multi-Agent / Concurrency Solution: If the model has a single-agent context limit of (e.g. 32.8k), but the physical cache capacity is larger (e.g. 81.9k), you can utilize the maximum context capacity by running multiple concurrent agents. vLLM automatically allocates memory blocks from the shared KV cache pool across parallel requests.
                

                

                    How to optimize: To increase usable context for a single agent/request, increase max-model-len in Context Settings. To run multiple concurrent agents to utilize the full KV cache pool, adjust concurrency Max Number of Sequences and deploy multiple client agents sharing the vLLM server endpoint.
